Claude Code プロジェクト管理 CLAUDE.md Skills 設定階層

TL;DR

Claude Codeのプロジェクト管理は「グローバル（~/.claude/）で共通基盤、各プロジェクト（.claude/）で固有設定」の2層構造が最適解。Skills/Agentsは親ディレクトリ走査に制限があるため、共通Skillsは~/.claude/skills/に集約し、プロジェクト固有は各.claude/skills/に配置する。スケーラビリティはテンプレート戦略とProgressive Disclosureで確保する。

設定階層の仕組み ― 2層共存の設計思想

Claude Codeには5段階のスコープが存在する: Managed > User > Project > Local > 子ディレクトリ。1Claude Code 公式ドキュメント - Settings この階層構造は、グローバル共通設定とプロジェクト固有設定の共存を設計思想として内包している。

重要なポイントは、CLAUDE.mdが親ディレクトリを遡って全て読み込む（フルロード）という点だ。つまり~/.claude/CLAUDE.mdに全プロジェクト共通ルールを書き、./CLAUDE.mdにプロジェクト固有ルールを書けば、両方が自動的にロードされる。1Claude Code 公式ドキュメント - Settings

settings.jsonのマージ動作

settings.jsonの配列設定（permissions.allow/deny等）はスコープ間でマージ・重複排除される。2Claude Code 公式ドキュメント - Configuration また@path/to/fileインポートで外部ファイル参照が可能で、最大5段階のネストに対応している。

Skills配置戦略 ― 親ディレクトリ非走査という制約

CLAUDE.mdとsettings.jsonは親ディレクトリを走査するが、skills/agents/commands/はCWD直下の.claude/のみ参照する。3GitHub Issue #26489 - Skills親ディレクトリ走査 この非対称性がSkills配置戦略を決定する最大の制約条件だ。

具体的には、Skills/Agentsは以下の2択になる:

  • ~/.claude/skills/: 全プロジェクト共通（commit, review, research等の汎用Skills）
  • .claude/skills/: プロジェクト固有（そのPJに特化したSkills）

「マスタープロジェクトにSkillsを集約して各プロジェクトから参照する」というパターンは、現時点では実現できない。GitHub Issue #26489（2026年2月提出）でFeature Requestされているが未実装だ。3GitHub Issue #26489 - Skills親ディレクトリ走査
